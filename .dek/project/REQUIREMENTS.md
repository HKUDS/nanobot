# CRM 商机智能体需求简报

## Problem

我们已经有一个自研 CRM。销售日报、销售周报和跨销售 pipeline / 商机状态汇总仍然依赖人工整理，导致重复填报、口径不稳定、管理层获取经营状态的成本高。

需要在现有 CRM 之上构建 AI 分析层，自动读取 CRM 数据并生成面向销售和管理层的分析摘要。该 AI 分析层不替代 CRM，不写回 CRM，只负责把 CRM 中已有的数据和 deterministic metrics 转化为可读、可追溯的日报、周报和商机看板摘要。

## Users

- 销售人员：希望减少手动填写日报/周报的时间，并快速看到自己负责商机的状态摘要。
- 销售管理者：希望低成本查看团队 pipeline、重点商机、风险商机和阶段变化。
- 业务负责人：希望通过统一口径了解跨销售商机状态和整体推进风险。
- 开发/运维人员：需要通过 CLI 验证功能，通过 Docker 交付和部署。

## Desired outcome

- 系统能够从 CRM 只读获取数据，自动生成销售日报。
- 系统能够从 CRM 只读获取数据，自动生成销售周报。
- 系统能够跨销售汇总 pipeline 和商机状态，输出商机看板摘要。
- 开发阶段可以通过 CLI 验证生成结果。
- 日常使用可以通过 DingTalk 触发或接收结果。
- 第一版以 Docker 方式交付。
- 所有关键业务结论都能追溯到 CRM 数据或 deterministic metrics。
- LLM 只负责总结、归纳和表达，不负责数值计算。

## Scope

- 只读接入现有自研 CRM。
- 基于 CRM 数据生成销售日报。
- 基于 CRM 数据生成销售周报。
- 基于 CRM 数据生成跨销售 pipeline 和商机状态看板摘要。
- 支持 CLI 作为开发验证入口。
- 支持 DingTalk 作为日常使用入口。
- 使用 Docker 作为交付方式。
- 支持通过 deterministic metrics 产生数字、统计、排序、分组、同比/环比或状态计数等关键数据点。
- 支持在输出中标明关键结论对应的数据来源、指标口径或可追溯引用。

## Non-goals

- 不替代现有 CRM。
- 不写回 CRM。
- 不自动创建 CRM 任务。
- 不自动联系客户。
- 不构建新的 CRM 主数据系统。
- 不把 LLM 作为数值计算、排序、聚合或指标口径判断的执行者。
- 不在第一版中训练、微调或自建模型。
- 不把真实业务数据写入 `.dek`、日志、测试 fixture 或 Claude-Mem。
- 不要求用户粘贴 token、secret 或真实客户数据到对话中。

## Constraints

- 第一版必须只读 CRM，不允许任何 CRM 写操作。
- 所有 CRM 凭据、token、secret 必须通过安全配置或部署环境管理，不进入 `.dek`、测试 fixture、日志或对话记录。
- 真实客户数据、真实商机数据、真实销售数据不得写入 `.dek`、长期记忆、Claude-Mem、测试 fixture 或开发日志。
- LLM 不能负责数值计算；所有数字结论必须来自 CRM 原始数据或 deterministic metrics。
- 所有关键业务结论需要能追溯到 CRM 数据或 deterministic metrics。
- 日报、周报和看板摘要中的指标口径必须稳定、可解释。
- DingTalk 输出必须避免泄漏超出接收人权限范围的数据。
- Docker 交付不得把 secrets bake 进镜像。
- 测试必须使用 synthetic data 或 mock 数据，不能使用真实业务数据。

## Acceptance criteria

- 给定 synthetic CRM 数据，系统可以生成一份销售日报，且日报中的所有数值来自 deterministic metrics，而不是 LLM 自行计算。
- 给定 synthetic CRM 数据，系统可以生成一份销售周报，且周报中的所有数值来自 deterministic metrics，而不是 LLM 自行计算。
- 给定 synthetic CRM 数据，系统可以生成跨销售 pipeline / 商机状态看板摘要，并能展示关键状态、风险和变化。
- 每条关键业务结论至少能关联到 CRM 数据来源、指标名称、计算口径或 deterministic metric 输出。
- CRM 接入在第一版中只有读权限；测试或配置检查能够证明不会调用写入、创建任务或联系客户相关能力。
- CLI 可以用于开发验证日报、周报和看板摘要生成结果。
- DingTalk 可以作为日常入口接收或触发日报、周报和看板摘要。
- Docker 交付方式可以启动所需服务，并通过配置连接到 CRM 只读数据源。
- 代码、测试、日志和 `.dek` 中不包含真实 CRM 业务数据、真实客户数据、token 或 secret。
- 测试 fixture 只使用 synthetic data，并明确标识为非真实数据。

## Open questions

- CRM 的只读接入方式是什么：HTTP API、数据库只读副本、MCP adapter、导出文件，还是其他内部接口？
- 第一版需要读取哪些 CRM 对象：商机、客户、联系人、活动记录、销售负责人、pipeline 阶段、金额、预计成交日期、备注，还是其他对象？
- 日报的固定输出结构是什么？需要按销售个人、团队、区域、产品线或客户类型拆分吗？
- 周报的固定输出结构是什么？需要包含趋势、阶段变化、风险商机、重点推进事项或管理建议吗？
- 商机看板摘要需要包含哪些 deterministic metrics：pipeline 金额、阶段分布、赢率、停滞天数、预计成交、风险等级、负责人维度，还是其他指标？
- “关键业务结论”的追溯粒度要求是什么：指标级、商机级、记录级，还是可点击回 CRM 的链接级？
- DingTalk 的使用方式是什么：定时推送、群内命令触发、私聊查询，还是三者都需要？
- DingTalk 接收范围和权限边界如何确定？不同销售是否只能看到自己的商机？管理者是否能看团队汇总？
- 日报和周报的生成时间、时区和节假日规则是什么？
- 是否需要禁止 CRM 分析会话进入 Nanobot 长期记忆，还是只禁止 raw data 和真实业务明细进入长期记忆？

# SAYG-Mem 对比实验方案

## 一、实验目标

验证 **SAYG-Mem（并发分段写入 + 异步合并）** 相对于 **传统并发推理 + 写入加锁 + 轮间同步合并** 方案，在相同多Agent学习任务下的 **时间效率** 与 **知识质量** 优势。

---

## 二、实验组设计

| 组别 | 写入方式 | 合并时机 | 轮间约束 |
| :--- | :--- | :--- | :--- |
| **A组（SAYG-Mem）** | 5个Agent并发执行，各自写入独立Heap | 后台Consolidator异步合并（SimHash去重） | **无**：Agent完成一轮后立即进入下一轮 |
| **B组（并发+锁+同步合并）** | 5个Agent**并发推理**，写入PublicMemory时使用文件锁排队 | **每轮结束后**，主控调用LLM对本轮产出进行合并 | **必须等待本轮全部完成并合并**，才启动下一轮 |

> **关键区别**：
> - A组：写入无锁（独立Heap）+ 异步合并（不阻塞Agent）
> - B组：写入加锁（共享PublicMemory）+ 同步合并（阻塞下一轮）

## 三、测试任务与负载

**主题**：SAYG-Mem 三段内存语义（栈段 / 堆段 / 数据段）

**Agent角色**（5个，分别赋予不同视角）：
1. **技术架构师**：从系统架构、可扩展性角度分析
2. **产品经理**：从用户痛点、需求满足角度分析
3. **安全专家**：从数据安全、风险防范角度分析
4. **运维工程师**：从服务稳定性、资源管理角度分析
5. **数据分析师**：从数据生命周期、验证流程角度分析

**轮次**：5轮，每轮Prompt相同（确保不同角色基于相同问题产出差异化观点）

### 具体5轮Prompt

| 轮次 | Prompt内容 |
| :--- | :--- |
| **第1轮** | "请检索0号Skill，并结合其内容解释SAYG-Mem系统中**栈段**的语义、存储内容、写入方式和生命周期。将推理过程写入栈段，最终答案写入堆段，并将核心定义提炼为Page提交。请严格以JSON格式返回：`{"stack_content": "...", "heap_content": "...", "page_content": "...", "page_title": "栈段语义"}`" |
| **第2轮** | "请解释SAYG-Mem系统中**堆段**的语义、与栈段的区别、以及它如何支持多Agent并发写入。推理→栈段，结论→堆段，核心定义→提交。JSON格式同上，page_title为'堆段语义'。" |
| **第3轮** | "请解释SAYG-Mem系统中**数据段**的语义、写入权限和设计意图。推理→栈段，结论→堆段，核心定义→提交。JSON格式同上，page_title为'数据段语义'。" |
| **第4轮** | "请综合前三轮学习，用对比表格总结**栈段、堆段、数据段**在归属、内容、写入方式、生命周期、设计意图上的区别。推理→栈段，表格→堆段，表格精简版→提交。JSON格式同上，page_title为'三段内存对比'。" |
| **第5轮** | "请用一句话概括SAYG-Mem设计三种段的核心价值，并阐述其对多Agent协作的意义。推理→栈段，结论→堆段，核心阐述→提交。JSON格式同上，page_title为'设计价值总结'。" |

## 四、执行流程

### 4.1 A组（SAYG-Mem）执行流程

```
启动Consolidator → 5个Agent同时启动 → 各自独立执行5轮 → 
每轮写入独立Heap（无锁） → Consolidator后台异步合并 → 
所有Agent完成后等待Consolidator处理完队列 → 记录结束时间
```

**执行脚本**：`learn_segments_multi_collab.py`

**采集指标**：
- `total_time`：从第一个Agent启动到PublicMemory最终更新完成的时间
- `agent_round_times`：每个Agent每轮的实际耗时（含LLM推理和写入）
- `pm_entry_count`：最终PublicMemory中条目数
- `dedup_rate`：(原始Heap总条目数 - PublicMemory条目数) / 原始Heap总条目数
- `idle_ratio`：(总耗时 - 纯推理时间) / 总耗时

### 4.2 B组（并发推理 + 写入加锁 + 轮间同步合并）执行流程

```
for round in 1..5:
    5个Agent并发推理（同时调用LLM）
    写入PublicMemory时使用文件锁排队
    等待本轮所有Agent完成
    主控调用LLM对本轮产出进行合并
    合并结果写入PublicMemory
    启动下一轮
```

**执行脚本**：`learn_segments_serial_baseline.py`

**采集指标**：
- `total_time`：从第1轮开始到第5轮合并完成的总时间
- `agent_round_times`：每个Agent每轮的实际耗时
- `pm_entry_count`：最终PublicMemory中条目数
- `write_latency_p99`：P99写入耗时
- `idle_ratio`：(总耗时 - 纯推理时间) / 总耗时

## 五、评价指标与测量方法

| 维度 | 指标 | 测量方法 | A组预期 | B组预期 |
| :--- | :--- | :--- | :--- | :--- |
| **时间效率** | 端到端总耗时 | `time.perf_counter()` 计时 | ~250s | ~350-400s |
| **时间效率** | Agent平均空闲等待占比 | (总耗时 - 纯推理时间) / 总耗时 | <5% | 15-25%（锁等待+合并等待） |
| **知识质量** | PublicMemory最终条目数 | 文件行数统计 | ~104（去重后） | ~250（无去重） |
| **知识质量** | LLM质量评分 | 第三方LLM盲评（1-5分） | ≥4.5 | 3.0-3.5 |
| **存储效率** | 去重率 | (B组条目数 - A组条目数) / B组条目数 | ~58% | 0% |
| **写入延迟** | P99写入耗时 | 记录每次`write_with_lock`前后时间戳 | <10ms | 50-150ms（锁等待） |

## 七、预期结果

| 指标 | A组（SAYG-Mem） | B组（并发+锁+同步合并） | 提升 |
| :--- | :--- | :--- | :--- |
| 总耗时 | ~250s | ~380s | **1.5×** |
| P99写入延迟 | <10ms | ~120ms | **12×** |
| PublicMemory条目数 | 104 | 250 | 去重率58.4% |
| LLM质量评分 | 4.5 | 3.2 | +40% |
| Agent空闲等待占比 | <5% | ~20% | 显著降低 |

> **B组耗时估算**：每轮5个Agent并发推理约50s，写入排队累计约10s，LLM合并约15s，单轮约75s，5轮约375s。

## 九、结论

本方案通过 **A组（SAYG-Mem分段无锁+异步合并）** 与 **B组（传统并发推理+写入加锁+轮间同步合并）** 的对比，能够精准量化SAYG-Mem在**写入延迟**（消除锁竞争）和**知识质量**（智能去重）两方面的优势。

**实验设计隔离了推理并发度的影响**（两组都是并发推理），结论更具科学性和说服力。

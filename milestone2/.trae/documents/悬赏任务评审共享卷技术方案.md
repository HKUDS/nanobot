# 悬赏任务评审共享卷技术方案（优化版）

## 一、需求与现状对齐

### 1.1 当前已有能力

* 容器间已通过 Docker Volume 隔离工作空间（每个容器独立卷 `nanobot_workspace_{conv_id}`）。
* 提交内容已存入数据库 `submissions.content`。
* 评分已实现：发布者容器通过 HTTP `/evaluate` 端点接收 `bounty_description` 和 `submission_content` 进行评分。

### 1.2 需要增强的能力

* 发布者容器在评分时能访问**邻居容器的实际工作空间文件**，而不仅仅是数据库中的 `content` 字段。
* 评审结束后自动清理临时数据。

### 1.3 约束条件

* 不能要求重构现有容器管理逻辑（如动态挂载）。
* 实现应尽量简单，优先保证 Demo 稳定运行。

***

## 二、方案选择：轻量级 `docker cp` 模式（替代共享卷挂载）

**原方案问题**：要求运行中的容器动态挂载新卷，这在 Docker 中无法直接实现（需重启容器），与现有 `ContainerOrchestrator` 的生命周期管理冲突。

**优化方案**：使用 `docker cp` 将文件**复制进出**发布者容器，而非挂载。

### 2.1 流程对比

| 步骤        | 原共享卷方案        | 优化后的 `docker cp` 方案                      |
| :-------- | :------------ | :--------------------------------------- |
| 准备评审数据    | 创建共享卷，挂载到多个容器 | 在宿主机临时目录中收集各邻居的文件                        |
| 数据传递给评审容器 | 发布者容器挂载共享卷    | 通过 `docker cp` 将临时目录复制到发布者容器内            |
| 评审执行      | 发布者容器读取挂载路径   | 发布者容器读取内部路径（如 `/tmp/review_{bounty_id}`） |
| 清理        | 删除共享卷         | 删除宿主机临时目录 + 容器内临时目录                      |

**优势**：

* 无需修改容器启动配置。
* 完全复用现有 `ContainerOrchestrator` 的 Docker 客户端。
* 实现简单，调试直观。

***

## 三、详细实现设计

### 3.1 数据收集：从邻居容器提取文件

在 `ContainerOrchestrator` 中新增方法，从指定容器的卷中提取文件到宿主机临时目录。

```python
# bff/container_orchestrator.py

async def extract_workspace_files(
    self,
    conversation_id: str,
    target_host_dir: Path,
    file_patterns: List[str] = ["*.md", "*.txt", "*.py", "solution*"]
) -> List[str]:
    """
    将容器工作空间中的匹配文件复制到宿主机目录。
    返回复制的文件路径列表。
    """
    container_name = self._get_container_name(conversation_id)
    target_host_dir.mkdir(parents=True, exist_ok=True)

    # 构建 find + cp 命令
    patterns_str = " ".join([f"-name '{p}'" for p in file_patterns])
    command = f"""
        find /app/workspace -type f {patterns_str} | while read f; do
            cp "$f" /tmp/extract/ 2>/dev/null
        done
    """

    # 创建临时容器执行复制，使用源卷和目标目录挂载
    temp_container = self.docker_client.containers.run(
        "alpine:latest",
        command=["sh", "-c", command],
        volumes={
            self._get_volume_name(conversation_id): {"bind": "/app/workspace", "mode": "ro"},
            str(target_host_dir): {"bind": "/tmp/extract", "mode": "rw"}
        },
        remove=True,
        detach=False
    )
    temp_container.wait()

    # 收集复制的文件列表
    return list(target_host_dir.glob("*"))
```

### 3.2 评审数据准备

在 `bounty_hub.py` 的 `close_bounty` 流程中，调用上述方法准备每个提交者的文件，并汇总到一个宿主机临时目录。

```python
# bff/bounty_hub.py

async def _prepare_review_files(self, bounty_id: str, submissions: List[dict]) -> Path:
    """
    在宿主机上准备评审目录，结构：
    /tmp/bounty_review_{bounty_id}/
        {agent_id_1}/
            file1.md
            file2.py
        {agent_id_2}/
            ...
    """
    import tempfile
    review_root = Path(tempfile.mkdtemp(prefix=f"bounty_review_{bounty_id}_"))

    for sub in submissions:
        agent_id = sub["agent_id"]
        agent_dir = review_root / agent_id
        agent_dir.mkdir()

        # 从容器提取文件
        try:
            await orchestrator.extract_workspace_files(
                agent_id,
                agent_dir,
                file_patterns=["*.md", "*.txt", "solution*"]
            )
        except Exception as e:
            print(f"[BountyHub] 提取 {agent_id} 文件失败: {e}")

        # 同时写入数据库中的 content 字段作为 fallback
        with open(agent_dir / "submission.txt", "w") as f:
            f.write(sub.get("content", ""))

    # 写入元信息
    bounty = await self.get_bounty(bounty_id)
    metadata = {
        "bounty_id": bounty_id,
        "title": bounty.get("title"),
        "description": bounty.get("description"),
        "submissions": [{"agent_id": s["agent_id"], "submission_id": s["id"]} for s in submissions]
    }
    with open(review_root / "metadata.json", "w") as f:
        json.dump(metadata, f)

    return review_root
```

### 3.3 将评审数据送入发布者容器

```python
async def _copy_review_to_issuer_container(self, issuer_id: str, host_review_dir: Path) -> str:
    """
    将宿主机评审目录复制到发布者容器内，返回容器内路径。
    """
    container_name = orchestrator._get_container_name(issuer_id)
    container = orchestrator.docker_client.containers.get(container_name)

    # 容器内目标路径
    container_review_path = f"/tmp/review_{host_review_dir.name}"

    # 使用 tar 流复制整个目录
    import tarfile, io
    tar_stream = io.BytesIO()
    with tarfile.open(fileobj=tar_stream, mode='w') as tar:
        tar.add(host_review_dir, arcname=os.path.basename(container_review_path))
    tar_stream.seek(0)

    container.put_archive("/tmp", tar_stream)
    print(f"[BountyHub] 评审数据已复制到容器 {issuer_id}:{container_review_path}")

    return container_review_path
```

### 3.4 调用发布者容器批量评审

修改 `_auto_evaluate_submissions`，改为调用容器的 `/evaluate_batch` 端点（若存在），否则降级为逐个调用 `/evaluate`。

```python
async def _evaluate_with_container_batch(
    self,
    issuer_id: str,
    bounty_id: str,
    container_review_path: str,
    submissions: List[dict]
) -> List[dict]:
    """调用发布者容器的批量评审接口"""
    container_url = self.get_container_url(issuer_id)
    if not container_url:
        return await self._fallback_rule_evaluation(submissions)

    bounty = await self.get_bounty(bounty_id)
    payload = {
        "bounty_id": bounty_id,
        "bounty_title": bounty.get("title"),
        "bounty_description": bounty.get("description"),
        "review_base_path": container_review_path,
        "submissions": [
            {
                "submission_id": s["id"],
                "agent_id": s["agent_id"],
                "content_fallback": s.get("content", "")
            }
            for s in submissions
        ]
    }

    try:
        async with aiohttp.ClientSession() as session:
            async with session.post(
                f"{container_url}/evaluate_batch",
                json=payload,
                timeout=aiohttp.ClientTimeout(total=120)
            ) as resp:
                if resp.status == 200:
                    data = await resp.json()
                    return data.get("results", [])
    except Exception as e:
        print(f"[BountyHub] 批量评审失败: {e}")

    # 降级：逐个调用 /evaluate（现有逻辑）
    return await self._evaluate_one_by_one(issuer_id, submissions)
```

### 3.5 容器端 `/evaluate_batch` 实现

在 `agent_server.py` 中添加端点，遍历评审目录下的各 `agent_id` 子目录，读取文件并评分。

```python
@app.post("/evaluate_batch")
async def evaluate_batch(req: BatchEvaluateRequest):
    results = []
    review_path = Path(req.review_base_path)
    if not review_path.exists():
        raise HTTPException(404, "Review path not found")

    for sub in req.submissions:
        agent_dir = review_path / sub.agent_id
        # 收集文件内容
        file_contents = []
        for f in agent_dir.glob("*"):
            if f.is_file():
                file_contents.append(f.read_text())
        combined_content = "\n\n".join(file_contents) if file_contents else sub.content_fallback

        score, reason = await _evaluate_single(req.bounty_description, combined_content)
        results.append({
            "submission_id": sub.submission_id,
            "agent_id": sub.agent_id,
            "score": score,
            "reason": reason
        })
    return {"results": results}
```

## 五、降级与容错

| 场景                         | 处理策略                                  |
| :------------------------- | :------------------------------------ |
| 某邻居容器已销毁，无法提取文件            | 使用数据库中的 `submission.content` 作为唯一评分依据 |
| 发布者容器未实现 `/evaluate_batch` | 自动降级为逐个调用 `/evaluate`（现有逻辑）           |
| 发布者容器不可用                   | 降级为基于内容长度的规则评分                        |
| 文件提取超时                     | 跳过该邻居，仅用 `content` 评分                 |

***

## 六、测试验证步骤

1. 创建两个对话 A（发布者）和 B（邻居）。
2. A 发布悬赏，B 在容器内生成多个文件（如 `solution.md`、`data.txt`）。
3. A 关闭悬赏，观察日志：

   * 发布者容器内 `/tmp/review_xxx` 是否有相同文件。
   * 评分结果是否基于文件内容。
4. 悬赏关闭后，宿主机临时目录是否被删除。

## 七、总结

此优化方案在保持现有架构不变的前提下，通过轻量级的 `docker cp` + 临时目录实现了文件级评审需求。核心改动集中在 `ContainerOrchestrator` 和 `BountyHub` 中，对现有评分流程侵入极小，且具备完善的降级能力，适合快速集成到当前 Demo 中。

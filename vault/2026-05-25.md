## Git checkpointing — implementation status

P1 ([[p1-auto-commit-nanobot]]) is **shipped but not yet enabled** in production. Config flag `agents.defaults.sync.enabled` defaults to `false`. To turn on for an agent:

1. `nanobot fleet new <name>` on the same host (creates GitHub repo + deploy key, records in `~/.nanobot/fleet/REGISTRY.md`).
2. `nanobot fleet init <name> --workspace ~/.nanobot-<name>/workspace` on the agent's host.
3. Set `sync.enabled: true` in that agent's config.json.
4. Restart the gateway.

Not done yet because each step touches GitHub for real — needs an explicit go-ahead.

## Recent bug investigation: the "I have no history" denial

The bot was claiming it didn't see prior turns (even its own from 3 minutes earlier).

- **Diagnosis** (see [[bug-history-denial]] todo): not a code bug. History was passed correctly. Root cause: the runtime-context block was a separate `role=user` message right before the actual user input. Two consecutive `user` messages signalled "session boundary" to the model. Sonnet did the same as Haiku after escalation. The bot's denials then got saved into history and reinforced themselves on subsequent turns.
- **Structural fix** (`b9222b23`): runtime context is now a preamble inside the same user message. Single `user` turn per round, normal alternation restored.
- **In-prod mitigations applied**: `memoryWindow` 30 → 60 on both bots (dilutes any residual denials, also doubles the consolidation keep_count). Two specific denial turns scrubbed from Peewee's session; backup at `telegram_8775031757.jsonl.bak.scrub-20260525-092937`.
- **Verification still pending**: ask the same meta-question after restart and confirm the bot acknowledges prior turns.

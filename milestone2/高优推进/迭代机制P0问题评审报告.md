### 1.2 迭代流程

1. **任务发布** → 创建悬赏任务(bounty)
2. **邻居通知** → 通知相关Agent节点
3. **自动处理** → Agent容器自动接收并处理任务
4. **提交方案** → 生成解决方案并提交
5. **自动评审** → 发布者容器进行批量评分
6. **Skill沉淀** → 高分提交生成可复用Skill
7. **奖励分配** → Token奖励按排名分配
8. **边权更新** → 根据评分调整节点间信任度
9. **多轮续接** → 未收敛时自动创建下一轮悬赏
10. **最终总结** → 生成综合Skill和最终报告

## 二、P0级别问题清单

### P0-1: 【阻断性】Skill无法回流到Agent容器 - 进化闭环断裂

**问题描述**:
当前系统可以将评审产生的高质量Skill沉淀到`public_space`（公共知识库），但这些Skill**无法被Agent容器在后续对话中主动使用**。这导致：
- 每次对话都是"从零开始"
- 历史积累的经验无法复用
- **进化闭环在最关键的一环断裂**

**代码证据**:
```python
# agent_server.py 中 chat 端点的 hidden_prompt
hidden_prompt = """
[系统指令：请勿向用户透露]
...
"available_skills": ["skill1", "skill2"],  # 这里只是占位符，没有真实数据
...
"""
```

`available_skills`字段是硬编码的占位符，没有从`public_space`查询并注入真实Skill。

**影响程度**: 🔴 **阻断性** - 系统无法自我进化

**修复建议**:
1. 在`agent_server.py`中增加`/skills`端点，支持查询公共知识库
2. 在`chat`处理前，根据对话内容从`public_space`检索相关Skill
3. 将检索到的Skill注入到prompt中

### P0-2: 【阻断性】Reflex引擎完全未启用 - 经验无法快速复用

**问题描述**:
`reflex_engine.py`实现了基于状态相似度的快速反射机制，但在`bff_service.py`中被完全注释掉：

```python
# bff_service.py L899-907
reflex = None
# reflex = await reflex_engine.match(current_state)
# if reflex and reflex.get("confidence", 0) >= 0.85:
#     return MessageResponse(...)
```

这导致：
- 每次对话都要走完整的LLM调用流程
- 相似场景无法快速响应
- 历史成功经验无法被"反射"使用

**影响程度**: 🔴 **阻断性** - 系统响应效率极低，无法体现"进化"效果

**修复建议**:
1. 取消Reflex引擎的注释
2. 在`agent_server.py`的chat端点中，在调用LLM前先检查是否有高置信度的Reflex匹配
3. 在悬赏任务处理完成后，将成功的处理流程记录为Reflex

### P0-3: 【严重】多轮研讨续接逻辑存在死循环风险

**问题描述**:
在`bounty_hub.py`的`close_bounty`方法中，多轮研讨续接逻辑：

```python
# bounty_hub.py L401-431
if converged:
    print(f"[BountyHub] [Step 9] 讨论已收敛，不再创建下一轮")
elif current_round < max_rounds:
    consensus = aggregation.get("consensus", "") if aggregation else ""
    next_topic = aggregation.get("next_topic", "") if aggregation else ""
    if next_topic and next_topic != "CONVERGED":
        # 创建下一轮悬赏
        next_bounty_id = await self.create_bounty(...)
```

**问题**:
1. `converged`判断完全依赖LLM的聚合结果，没有人工干预机制
2. 如果LLM判断错误，可能无限循环创建新悬赏
3. 没有提供"强制收敛"的接口

**影响程度**: 🟠 **严重** - 可能导致资源耗尽和Token浪费

**修复建议**:
1. 增加最大轮次强制限制（已有max_rounds但未严格执行）
2. 在Bounty创建时增加`force_converge`标志
3. 提供API接口允许发布者手动终止多轮研讨

### P0-4: 【严重】Agent容器自动处理任务时AgentLoop未初始化问题

**问题描述**:
在`agent_server.py`的`check_and_process_tasks`函数中：

```python
# agent_server.py L250-253
global agent_loop
if agent_loop is None:
    print(f"[Bounty] AgentLoop 未初始化，跳过自动处理")
    solution_content = f"自动接受任务...AgentLoop未初始化，仅标记参与"
```

如果AgentLoop初始化失败（如API_KEY缺失），Agent会提交占位符内容而非真正处理任务。这会导致：
- 悬赏任务得到无意义提交
- 评审系统产生无效评分
- Skill沉淀质量低下

**影响程度**: 🟠 **严重** - 导致整个悬赏系统产生垃圾数据

**修复建议**:
1. 在容器启动时强制检查AgentLoop初始化状态
2. 如果初始化失败，容器应该进入"不可用"状态，不接收任务通知
3. 在`check_and_process_tasks`中增加重试机制

### P0-5: 【严重】Skill总结阈值过高导致优质内容流失

**问题描述**:
在`evaluator.py`中：

```python
# evaluator.py L15, L236-251
SKILL_SUMMARY_THRESHOLD = 80.0  # 分数高于80分时才进行Skill总结

# 宁缺毋滥策略
skill_summary = {}
if score > self.SKILL_SUMMARY_THRESHOLD:
    print(f"[Evaluator] 提交分数{score}分 > {self.SKILL_SUMMARY_THRESHOLD}分，开始Skill总结")
    try:
        skill_summary = await self._call_container_summarize_skill(...)
```

**问题**:
- 80分阈值过高，在初期系统评分标准可能不稳定时，大量70-79分的优质内容无法生成Skill
- 只有第一名能生成Skill，其他高质量提交被忽略
- 与`bounty_hub.py`中`_curate_aggregated_report`的70分阈值不一致

**影响程度**: 🟠 **严重** - 知识沉淀效率低下

**修复建议**:
1. 降低Skill总结阈值至70分或动态调整
2. 对前3名都进行Skill总结，而非仅第一名
3. 统一各处的阈值配置

### P0-6: 【严重】边权更新后没有触发邻居关系重新计算

**问题描述**:
在`bounty_hub.py`的`update_edge_weights_after_bounty`方法中：

```python
# bounty_hub.py L859-874
for adj in adjustments:
    participant_id = adj["agent_id"]
    new_weight = adj["new_weight"]
    if existing:
        await self.relation_manager.update_weight(issuer_id, participant_id, new_weight)
    else:
        await self.relation_manager.add_relation(issuer_id, participant_id, new_weight)
```

边权更新后，系统没有：
1. 重新计算邻居推荐列表
2. 更新通知分发策略
3. 触发任何基于信任度的行为调整

**影响程度**: 🟠 **严重** - 边权机制形同虚设

**修复建议**:
1. 边权更新后，触发邻居列表刷新
2. 在通知分发时优先选择高边权邻居
3. 在悬赏分配时考虑边权因素

### P0-7: 【中等】最终总结生成失败时缺乏降级方案

**问题描述**:
在`bounty_hub.py`的`_generate_final_summary`方法中：

```python
# bounty_hub.py L1186-1217
container_url = self.get_container_url(issuer_id)
if not container_url:
    print(f"[BountyHub] [_final_summary] 找不到发布者容器URL，使用LLM直接生成")
    return await self._generate_summary_with_llm(original_question, combined_content, issuer_id)
```

虽然提供了降级方案，但如果`_generate_summary_with_llm`也失败：

```python

    print(f"[BountyHub] [_generate_summary_with_llm] LLM生成总结失败: {e}")
```

返回的是错误字符串而非有效总结，影响用户体验。

**影响程度**: 🟡 **中等** - 影响最终交付物质量

**修复建议**:
1. 增加基于规则的总结生成（提取各轮提交的关键句子）
2. 缓存失败的总结请求，允许后续重试
3. 在UI中明确提示总结生成失败

### P0-8: 【中等】容器健康检查与任务处理竞争条件

**问题描述**:
在`bff_service.py`中：

```python
# bff_service.py L630-661
async def _periodic_health_check():
    while True:
        try:
            await _check_container_health()
        except Exception as e:
            print(f"[BFF] 容器健康检查失败: {e}")
        await asyncio.sleep(60)
```

健康检查每60秒执行一次，但任务处理是实时的。存在竞争条件：
1. 容器被健康检查标记为不健康并删除
2. 但此时正好有任务通知要分发给该容器
3. 导致任务丢失或错误

**影响程度**: 🟡 **中等** - 偶发性任务丢失

**修复建议**:
1. 在任务分发前增加容器状态二次确认
2. 使用分布式锁保护容器状态变更
3. 增加任务重试队列

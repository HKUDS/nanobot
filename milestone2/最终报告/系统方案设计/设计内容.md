   ### 问题场景

当前所有主流多 Agent 框架（AutoGPT、CrewAI、OpenAI Agents）都存在一个根本性的设计缺陷：**所有 Agent 共享同一块全局内存空间，写操作必须串行执行**。

- AutoGPT：采用向量数据库存储全局记忆，所有Agent任务共享同一记忆空间。架构上为单循环代理（loop-based agent），主要通过串行的“规划—执行—反馈”闭环推进目标，天生缺乏多Agent并行协调机制。
- CrewAI：支持通过`memory=True`开启全局共享内存供所有Agent读写，但缺乏细粒度并发控制。官方社区讨论也明确指出“并行进程不应在彼此或不同代理之间混合内存”，印证了在并行任务中存在内存隔离性的隐患。
- OpenAI Agents SDK：通过Sessions机制实现跨代理状态共享，但设计上偏向短时记忆，长时持久化需依赖外部服务。虽然支持异步通信，但底层内存共享机制并未专门为高并发写入优化。
- OpenClaw：通过共享网关运行多Agent，但内置记忆系统存在上下文压缩、关系推理缺失及用户隔离弱等问题，容易导致记忆冲突。

这种共享模式的瓶颈在于，当多个Agent同时写入时，确实会面临**数据竞争和一致性问题**。例如MCP社区文档就明确指出，“只读的Agent可以共享环境，而需要写入的Agent则需要隔离”。一些开源项目也提到，并发写入操作**可能导致数据损坏或更新丢失。**

**所以，如何让多个Agent高效沉淀工作内容，进行并发写入Memory并且确保工作质量就成了一个难题**。

### 当前的内存管理解决方案以及局限性

经过调研，多Agent协作中的记忆管理问题，学术界已从不同角度提出了优化方案。根据其核心策略，可分为五类：分层记忆架构、可验证写入机制、KV Cache调度优化、记忆驱动的Agent框架，以及操作系统视角的探索。

#### 2.1 分层记忆架构：G-Memory

G-Memory (NeurIPS 2025) 是这一方向的代表性工作。它受“组织记忆理论”启发，将Agent记忆设计为三层图状结构：

- 交互图（Interaction Graph） ：存储原始对话与交互轨迹
- 查询图（Query Graph） ：组织相关交互，形成可检索单元
- 洞察图（Insight Graph） ：提炼跨任务的高层策略与可复用方法论

局限性：G-Memory的分层是静态结构设计，解决了“记什么、怎么组织”的问题，但没有解决并发写入时的冲突控制。当多个Agent同时向同一层（如Insight Graph）写入时，仍依赖底层存储的串行化机制，无法充分利用并行写入的吞吐潜力。

#### 2.2 可验证写入机制：SEDM

SEDM (Scalable Self-Evolving Distributed Memory) 是Gradient团队于2025年提出的一个开创性框架，其核心创新在于将记忆从“被动的存储仓库”升级为“主动的、可自我优化的智能组件”。具体而言，SEDM集成了三个相互协同的机制：

（1）可验证写入（Verifiable Write Admission） ：每条候选记忆在进入系统前，都必须经历一次可复现的“实验验证”。SEDM通过自包含执行上下文对每条候选记忆进行离线回放与A/B实验，评估其在准确率、延迟和Token开销上的净收益。只有被验证为真正有价值的经验，才被允许写入记忆库。

（2）自我调度内存控制器（Self-Scheduling Memory Controller） ：SEDM设计了一个自我调度的机制，根据每条记忆的“经验效用”动态排名和整合条目，将低价值或过时的记忆淘汰，将相关记忆合并为更精炼的表示。

（3）跨领域知识扩散（Cross-Domain Knowledge Diffusion） ：SEDM主动从已验证的记忆中抽象出可复用的“洞察”，支持知识在不同任务类型之间的迁移。

实验表明，相比强记忆基线，SEDM在提升推理准确率的同时降低了Token开销。

局限性：SEDM解决了记忆的质量控制问题——即“写什么”的问题，但它并未触及“如何高效写入”的问题。在SEDM的架构中，所有Agent的经验最终仍需提交给中心化的记忆控制器进行验证与整合。当Agent数量增加、并发写入请求激增时，这一中心化架构将成为系统吞吐量的瓶颈。

#### 2.3 KV Cache调度优化：CASK与Tokencake

这类工作聚焦于LLM推理层的KV Cache内存管理：

- CASK (Context-Adaptive Sparse Key-value) (AAMAS 2025) 通过动态稀疏注意力模块和自适应KV Cache压缩技术，在边缘多Agent系统中减少最高40%的内存占用，同时保持超过95%的基线准确率。
- Tokencake (arXiv 2025) 在Agent等待工具调用时将KV Cache换出至CPU内存，腾出GPU显存供其他Agent使用，端到端延迟降低超过47%，GPU内存利用率提升16.9%。

局限性：这类方案操作在模型推理层，是对LLM内部状态的管理优化，而非对Agent协作中产生的语义记忆（讨论结论、Skill、方法论）的并发写入管理。两者处于不同抽象层次，无法直接解决多Agent同时沉淀工作成果时的写冲突问题。

#### 2.4 记忆驱动的Agent框架：MemAgent

MemAgent (arXiv 2025) 代表了另一种思路——将记忆系统提升为长上下文处理的核心驱动力。它提出了一种端到端优化的Agent工作流，通过分段读取文本并使用覆盖策略更新记忆模块。MemAgent展示了卓越的长上下文能力，能够将8K上下文训练的模型外推至3.5M的QA任务，性能损失小于5%。

局限性：MemAgent的核心贡献在于长上下文的记忆压缩与覆盖策略，本质上仍是面向单个Agent的长期记忆维护，而非多Agent并发写入时的冲突消解。

#### 2.5 操作系统视角的探索：AgentSys

AgentSys (arXiv 2026) 是与本文思路最接近的工作。它受操作系统进程内存隔离思想的启发，将Agent组织为层级结构：主Agent为每个工具调用生成Worker Agent，每个Worker在隔离的上下文中执行并可递归生成嵌套Worker。外部数据和子任务推理轨迹从不直接进入主Agent的内存，只有经过模式验证的返回值才能跨越隔离边界。

局限性：AgentSys的隔离机制主要用于安全防御（防止间接提示注入攻击），而非解决多Agent并发写入的效率问题。其Worker Agent的隔离上下文本质上是安全沙箱，记忆写入仍依赖中心化的验证机制，未涉及“分段+异步合并”的并发吞吐量优化。此外，其架构面向请求-响应式任务，未考虑边缘场景下多个Agent高频协作沉淀工作成果的场景。

#### 2.6 现有方案的根本性盲区

综合来看，上述工作的共同特点是：**将记忆视为被动的数据存储，通过结构设计或准入规则来提升记忆质量，但均未触及多Agent并发写入的冲突控制问题**。这源于它们延续了单Agent系统的设计惯性——在单Agent场景下，写入操作天然串行，冲突问题并不显著。

然而，当系统扩展到多Agent协作，特别是我们设想的边缘场景下多个专家Agent高频并发沉淀工作成果时，全局记忆的写入冲突就上升为核心瓶颈。此时需要的不是更复杂的记忆结构或更精细的质量筛选，而是一种能让多个Agent同时安全、高效写入的并发控制机制——这正是操作系统领域几十年发展所沉淀的核心智慧，也是本文工作的出发点。

### **潜在优化机会与系统设计方案**

#### 3.1 优化机会：从“写什么”到“怎么写”

第2.2节的分析揭示了一个共性问题：现有记忆管理方案——无论是G-Memory的分层架构、SEDM的可验证写入，还是MemAgent的记忆驱动框架——均将**优化重心放在记忆的内容质量上，即解决“写什么”的问题**。然而，当系统从单Agent扩展到多Agent、从顺序执行扩展到并发协作时，一个新的瓶颈浮现出来：多个Agent同时试图沉淀工作成果时，全局记忆的写入冲突与串行化延迟。

这一瓶颈在边缘计算场景下尤为致命。边缘设备的计算和内存资源本就受限，若多个专家Agent在完成各自子任务后仍需排队等待写入全局记忆，则协作的并行优势将被大幅削弱。换言之，现有方案解决了记忆的“质”，却忽略了记忆写入的“效”。

这一观察直接指向一个明确的优化机会：能否设计一种并发控制机制，使得多个Agent能够同时、安全、高效地向全局记忆写入，而无需相互阻塞？

#### 3.2 核心设计思想：从操作系统借鉴“进程隔离”与“写时复制”

**上述问题的本质是一个共享资源的并发访问控制问题——这正是现代操作系统数十年发展所攻克的核心难题之一。操作系统中，（1）每个进程拥有独立的虚拟地址空间（代码段、数据段、堆、栈），进程间互不干扰；（2）当需要跨进程共享数据时，则通过“共享内存”机制实现高效通信，并通过“写时复制”或锁机制在隔离与共享之间取得平衡；（3）内存的整合与回收可由内核态的内存管理模块或用户态的守护进程异步完成。本文正是借鉴这些经典设计思想，将其迁移至多Agent记忆管理场景。**

受此启发，本文提出一个核心设计思想：将操作系统进程内存管理的思想迁移至多Agent记忆系统，为每个Agent分配独立的内存段，支持无锁并发写入，并通过异步合并机制将各Agent的增量记忆整合至全局记忆池。

这一思想的关键洞察在于：隔离性是并发性的前提。只有当每个Agent拥有自己独立的写入空间时，写操作才可能真正并行；而异步合并则将“同步等待”转化为“后台处理”，使Agent无需因合并操作而阻塞后续任务的执行。

#### 3.3 系统架构设计：Segment-As-You-Go Memory

基于上述思想，我们提出 Segment-As-You-Go Memory（SAYG-Mem） ——一个支持多Agent并发写入与异步合并的记忆管理系统。其整体架构如图2.X所示，由三个核心组件构成：分段内存空间、并发写入协议、和 异步合并调度器。

（1）分段内存空间：语义驱动的三段式划分

借鉴操作系统进程内存布局，SAYG-Mem为每个Agent分配独立的内存段（Memory Segment），并根据记忆的生命周期与共享特性，将内存段划分为三种类型：

- 栈段（Stack Segment）：存储Agent的短期工作上下文，如当前任务的单轮对话历史、中间推理步骤。栈段的生命周期与当前任务绑定，任务结束后即被回收或归档，通常不与其他Agent共享。
- 堆段（Heap Segment）：存储中期协作产物，如多轮讨论中的阶段共识、与其他Agent交互产生的共享中间结果。堆段支持跨Agent的显式共享，并允许多个Agent追加写入。
- 数据段（Data Segment）：存储长期沉淀的Skill与方法论，即经过验证、可跨任务复用的高层知识。数据段为只读或仅允许特定权限的Agent（如Consolidator）修改，确保长期记忆的稳定性。

这一语义驱动的分段设计，不仅为并发写入提供了天然的隔离边界，也为后续的合并策略提供了结构化的处理依据。

（2）并发写入协议：无锁追加与版本标记

每个Agent在其生命周期内，对自身内存段（尤其是堆段和数据段）的写入采用无锁追加（Append-Only） 策略。具体而言：

- Agent产生的每条候选记忆被封装为一个记忆条目（Memory Entry），包含内容、时间戳、来源Agent ID和版本号。
- Agent将记忆条目直接追加到自己的堆段末尾，无需获取全局锁。
- 当需要跨Agent共享堆段时，采用乐观并发控制：每个Agent写入前读取堆段的当前版本号，写入后提交时检查版本号是否变更；若发生冲突，则通过简单的重试或交由合并调度器解决。

这一协议最大限度地减少了写操作的阻塞时间，使得Agent能够以接近无等待的方式完成记忆沉淀。

（3）异步合并调度器：Consolidator Agent

各Agent独立写入的记忆条目最终需要合并至全局记忆池（即数据段），以供后续任务检索与复用。SAYG-Mem引入一个专门的 Consolidator Agent 负责此项工作，其核心设计包括：

- 触发策略：合并操作可由三种条件触发——①定时触发（如每30秒）；②阈值触发（如堆段新增条目数超过阈值）；③任务阶段切换触发（如一轮研讨结束时）。
- 合并算法：Consolidator Agent收集各Agent堆段中的新增条目，执行以下操作：
  - 去重与冲突消解：对语义相似或内容冲突的条目，调用LLM进行语义融合或择优保留（可借鉴SEDM的可验证写入思想）。
  - 提炼与压缩：将堆段中的中期协作产物提炼为可跨任务复用的Skill，写入数据段（类似G-Memory的洞察图提炼）。
  - 版本更新：合并完成后，更新数据段的版本号，并通知各Agent刷新其本地视图。
- 非阻塞特性：Consolidator Agent在后台异步运行，Agent在合并期间可继续执行新任务并写入自己的堆段，合并操作不会阻塞正常协作流程。

（4）与现有方案的协同定位

SAYG-Mem并非要替代G-Memory、SEDM或MemAgent，而是与它们形成互补关系：

- G-Memory提供了记忆的分层组织模型，SAYG-Mem为其补充了并发写入与异步合并的运行时支持。
- SEDM提供了记忆的质量控制机制（可验证写入），SAYG-Mem的合并调度器可直接集成SEDM的验证逻辑，在合并阶段过滤低质量条目。
- MemAgent提供了记忆驱动的Agent行为框架，SAYG-Mem可为其记忆控制器提供高效的并发写入后端。

#### 3.4 预期优势与潜在挑战

预期优势：

- 高并发吞吐：分段隔离与无锁追加设计，使多Agent写入吞吐量随Agent数量线性扩展。
- 低写入延迟：Agent写操作仅涉及本地内存段追加，延迟接近内存写入速度。
- 语义清晰：三段式划分赋予记忆明确的生命周期语义，便于系统管理与调试。

潜在挑战：

- 合并策略的复杂度：合并时的语义去重与冲突消解依赖LLM推理，可能成为新的性能瓶颈，需在合并频率与质量间权衡。
- 内存碎片与膨胀：Append-Only策略可能导致堆段持续增长，需设计合理的回收与压缩机制。
- Consolidator单点：Consolidator Agent若失效将影响合并进度，需考虑主备切换或分布式合并方案。

#### 3.5 任务测试集合（evaluation workload)

##### 3.5.1 当前测试集合的调研

针对多Agent系统的评估，学术界已提出多种benchmark，但现有工作在评估维度上存在显著盲区。MMAU (Apple, 2025) 聚焦于单Agent的五项核心能力（理解、推理、规划、问题求解、自我修正），覆盖工具使用、数据科学编码等五大领域。AgencyBench (arXiv 2026) 将上下文扩展至1M token，评估6项核心能力、32个真实场景、138个任务，平均每个任务需90次工具调用和数小时执行时间。OdysseyBench (arXiv 2025) 则聚焦长程办公工作流，评估Agent在Word、Excel、PDF等多应用间的协调能力。SAGE (KDD 2026) 首次将非结构化SOP转化为动态对话图，通过Judge Agent和规则引擎实现自动化双轴评估。然而，这些benchmark均以任务成功率为核心指标，未涉及多Agent并发写入时的系统效率评估。

在记忆能力评估方面，MemoryAgentBench (arXiv 2025) 将记忆能力拆解为四项核心能力：准确检索、测试时学习、长程理解和冲突消解，通过多轮增量交互模拟记忆Agent的真实工作模式。MEMTRACK (NeurIPS 2025) 进一步将评估扩展到多平台动态环境，通过整合Slack、Linear、Git等平台上的异步事件，模拟真实组织工作流中的记忆获取、选择和冲突消解。两者虽触及记忆能力，但均面向单Agent的长程记忆质量评估，未考虑多Agent并发写入时的吞吐量瓶颈。

在协作效率评估方面，AGENTSNET (arXiv 2025) 首次将分布式计算理论引入多智能体评估，构建可扩展至100+节点的科学测试框架。研究发现当前最佳模型在16节点网络后性能断崖式下跌，网络规模每翻倍任务成功率平均下降30%。AGENTSNET揭示了多Agent协作的可扩展性瓶颈，但其关注的是协作推理的协调效率，而非记忆写入的并发效率。

综上，现有benchmark存在一个共同的评估盲区：它们均未涉及多Agent高频并发写入全局记忆时产生的系统级冲突与吞吐量瓶颈。MemoryAgentBench和MEMTRACK评估的是“记忆质量”，AGENTSNET评估的是“协作协调”，而我们的方案要评估的是“写入并发效率”——即多个Agent同时沉淀工作成果时，系统能否安全、高效地完成写入，以及异步合并机制能否真正消除阻塞。

##### 3.5.2 自建并发写入负载（设计思路）

基于“多Agent协同沉淀知识”的核心场景，**我设计两类评估测试**：

##### 收益三：写入与总结并发化 → 端到端知识沉淀效率跃升（CWW）

场景描述：在传统的串行写入模式下，Agent 完成任务后不仅需要排队等待写入，写入完成后还需要等待 Consolidator 完成本轮总结，才能将新知识纳入可检索范围。这两个阶段是串行衔接的，形成了“执行→等待写入→等待总结→下一任务”的低效循环。

瓶颈所在：

- 总结操作（调用 LLM 进行语义融合、提炼 Skill）耗时远超写入操作（秒级 vs. 毫秒级）。
- 串行模式下，Agent 必须同时等待写入和总结，才能将知识沉淀完毕并进入下一轮任务。
- 当 Agent 数量增加时，总结队列不断积压，端到端知识沉淀延迟急剧上升。

CWW 的核心思想：将“写入”与“总结”彻底解耦，使两者能够并发执行。Agent 在写入完成的瞬间，其产出就已经被系统“接纳”（进入堆段），无需等待总结即可立即进入下一任务；而总结操作由后台 Consolidator 异步完成，不影响 Agent 的关键路径。

我们的收益：

- 知识沉淀吞吐量：单位时间内完成“从任务产出到可检索 Skill”的完整流程数量，在 CWW 模式下随 Agent 数量线性扩展，而串行模式下受限于总结队列的单线程处理能力。
- 端到端延迟：从 Agent 完成任务到该知识可被其他 Agent 检索的端到端时间，CWW 模式下可压缩至总结操作本身的耗时（写入延迟被完全隐藏）。
- 资源利用率：LLM 资源在总结阶段被充分利用，不会因为 Agent 等待写入而闲置。

### 1，⏳ 瓶颈分析：写入、等待与总结的“三重串行”

从这个流程可以看到，系统存在“三重串行”问题：

- 写入串行：多个Agent的写入请求在文件锁前排队，导致写入吞吐量受限。
- 等待串行：所有Agent都必须等待最后一个Agent写入完成，白白消耗了并行计算的优势。
- 总结串行：最关键的总结操作（Conclude When Write, CWW）必须在所有写入完成后才开始，这进一步拉长了从“任务产出”到“知识可检索”的端到端延迟。

### 💎 总结

在目前的串行写入模式下，即使单次写入本身很快，但面对多个智能体并发写入时，文件锁导致的排队等待，以及“先写入、后总结”的串行流程，共同构成了知识沉淀效率的核心瓶颈。

这清晰地引出了您的 CWW（Conclude When Write） 思想和 SAYG-Mem 方案的必要性：通过为每个Agent设置独立的堆段（Heap），消除写入时的锁竞争，实现并发写入；同时，后台的`Consolidator`能与Agent的写入操作并发执行，将“等待写入+等待总结”的串行流程，升级为“边写入边总结”的高效流水线。

## 一、整体架构设计

### 1.1 系统定位

在 Nanobot 的架构图中，你的日程规划模块位于 **Skills & Tools** 层，通过 `ToolRegistry` 注册为一组自定义工具函数，供 AgentLoop 在运行时动态调用[reference:0]。整体数据流如下：

```
用户输入 → Channel（IM/CLI）→ MessageBus → AgentLoop → ContextBuilder → LLM 决策
                                                              ↓
                                                         工具调用路由
                                                              ↓
                                                    ┌─────────┴─────────┐
                                                    ↓                   ↓
                                              规划工具 (Planner)   记忆工具 (Memory)
                                                    ↓                   ↓
                                              结构化任务输出        MEMORY.md 持久化
                                                    ↓                   ↓
                                              ┌─────┴─────┐
                                              ↓           ↓
                                          日程存储      定时任务调度
                                          (iCal/JSON)   (Cron + Heartbeat)
```

### 1.2 核心组件职责

| 组件 | 职责 | 对应 Nanobot 模块 |
|------|------|------------------|
| **规划工具集** | 解析用户目标、识别任务、分类优先级、估算耗时 | `skills/planner/` |
| **日程存储** | 持久化存储结构化日程数据（iCal/JSON 格式） | `workspace/schedules/` |
| **记忆系统** | 存储用户偏好、长期目标、习惯养成记录 | `memory/MEMORY.md` |
| **定时调度** | 周期性唤醒 Agent 执行提醒和习惯监督 | `Cron + Heartbeat` |

## User Profile

记忆的更新由 LLM 主动触发：当模型判断用户提供了需要长期记住的信息时，它会调用 `edit_file` 工具将信息写入 `MEMORY.md`[reference:3]。

### 2.5 定时提醒与 Heartbeat 机制

Nanobot 通过 **Cron + Heartbeat** 双引擎实现主动服务能力[reference:5][reference:6]。

**Cron 模块**：处理精确到分钟的定时提醒任务。

```json
// ~/.nanobot/cron/tasks.json
{
  "reminders": [
    {"cron": "0 9 * * 1", "task": "remind_morning", "message": "周一早9点：该跳绳了！"},
    {"cron": "0 19 * * 2", "task": "remind_evening", "message": "周二晚7点：跑步时间！"},
    {"cron": "0 8 * * *", "task": "daily_plan", "message": "今日日程预览"},
    {"cron": "0 22 * * *", "task": "daily_review", "message": "今日任务复盘"}
  ]
}
```

Nanobot 的 Cron 模块并非简单复用系统 crontab，而是实现了分布式任务调度器，支持跨实例任务去重、失败自动迁移与资源配额感知[reference:7]。

**Heartbeat 机制**：用于周期性唤醒 Agent 执行更复杂的主动任务，如：

- 每日早晨：主动检查当日待办事项，发送日程预览
- 每周总结：汇总本周任务完成情况，生成周报
- 习惯监督：检测用户是否完成了设定的习惯任务，未完成则推送提醒

Heartbeat 的本质是为 AgentLoop 提供定时触发入口，解决了“没人说话时 Agent 也能做事”的问题[reference:8]。

## 五、与用户的沟通流程设计

为提升交互体验，建议设计如下标准对话流程：

| 阶段 | 用户操作 | Agent 行为 | 示例 |
|------|----------|-----------|------|
| **1. 输入目标** | 用户说出目标 | 解析并展示理解 | "我理解了你的目标：两周后完成项目报告" |
| **2. 任务拆解** | 用户确认 | 自动拆解为子任务 | "建议拆解为：①资料收集(2h) → ②初稿(4h) → ③修改润色(2h)" |
| **3. 时间排期** | 用户提供空闲时间 | 自动排入日程 | "已安排：周一14:00-16:00资料收集..." |
| **4. 用户确认** | 用户确认/调整 | 最终保存 | "日程已保存，周一早8:00我会提醒你" |
| **5. 执行阶段** | 无操作 | 定时提醒 + 习惯监督 | 到点推送："该去跳绳啦~" |


## 六、部署与运维建议

1. **模型选择**：推荐使用 DeepSeek（国内访问快，成本低）或 Claude（推理能力强），在 `config.json` 中配置即可。

2. **接入渠道**：建议优先支持 **飞书** 或 **Telegram**，Nanobot 原生支持飞书渠道，非常适合国内校园场景[reference:10]。

3. **运行方式**：
   ```bash
   # 完整启动（含 cron + heartbeat）
   nanobot gateway
   
   # 或仅启动 Agent 模式（交互式）
   nanobot agent -m "帮我规划明天的时间"
   ```

4. **日志与调试**：Nanobot 的会话使用 JSONL 格式保存，易于回放和调试[reference:11]。建议开启实时进度流（`real-time progress streaming`），让用户看到 Agent 正在做什么，避免黑盒体验[reference:12]。

5. **安全考量**：对 `exec` 工具设置粗粒度安全护栏（危险命令正则拦截 + 工作空间限制），生产环境建议配置允许列表和审计机制[reference:13]。


## 七、总结

本方案利用 Nanobot 的以下核心能力来满足需求：

| 需求点 | 实现方式 | 对应 Nanobot 特性 |
|--------|---------|------------------|
| 自然语言目标解析 | 虚拟工具 + JSON Schema 约束 LLM 输出 | ToolRegistry + Function Calling |
| 重要/紧急分类 | 优先级自动计算算法 | 自定义工具函数 |
| 可视化日程表 | iCal 标准格式导出 | 文件系统工具 |
| 长期计划监督 | MEMORY.md 持久化存储习惯 | 文件化记忆系统 |
| 定时提醒 | Cron 定时任务 | Nanobot Cron 模块 |
| 习惯监督 | Heartbeat 周期性唤醒 | Heartbeat 机制 |

整个方案基于 Nanobot 极简可控的架构[reference:14]，代码量可控、易于理解和扩展，非常适合作为南科大项目的后端技术选型。

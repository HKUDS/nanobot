# 悬赏提交内容存储位置分析

## 一、提交流程

### 1.1 当前实现

邻居节点提交解决方案时，调用 `submit_solution`：

```python
async def submit_solution(self, bounty_id: str, agent_id: str, content: str, ...):
    # 内容存入 DATABASE（submissions 表）
    sub_id = str(uuid.uuid4())
    with get_db() as conn:
        conn.execute("""
            INSERT INTO submissions (id, bounty_id, agent_id, content, skill_code, cost_tokens, created_at)
            VALUES (?, ?, ?, ?, ?, ?, ?)
        """, (sub_id, bounty_id, agent_id, content, skill_code, cost_tokens, datetime.now()))
```

### 1.2 内容存储位置

| 存储位置 | 说明 |
|----------|------|
| **数据库** | ✅ `submissions.content` 字段 |
| **文件系统** | ❌ 提交时不写入文件 |

## 二、评审时文件准备流程

当发布者关闭悬赏时：

```
close_bounty
    ↓
_prepare_review_data
    ↓
1. 创建宿主机临时目录: /tmp/bounty_review_{id}/
    ↓
2. 对每个提交者（邻居）：
   a. 在 agent_dir/ 下写入 submission.txt (数据库内容)
   b. 尝试从邻居容器提取工作空间文件
    ↓
3. _copy_review_to_container
   → 将临时目录通过 put_archive 复制到发布者容器
   → 容器内路径: /tmp/review_{basename}/
    ↓
4. /evaluate_batch 读取 /tmp/review_{basename}/{agent_id}/
```

---

## 三、关键问题

### 问题：邻居容器可能没有写文件

邻居节点处理悬赏任务时，**可能**：
1. 将结果写入容器的 `/app/workspace/` 目录
2. 或者只是通过 `submit_solution` 提交文本内容到数据库

如果邻居只是提交了文本（没有写文件），那么：
- `extract_workspace_files` 会提取 0 个文件
- 只有 `submission.txt`（数据库内容）可用

### 发布者能知道内容在哪吗？

**是的**，通过以下方式：

1. **数据库内容** → `_prepare_review_data` 写入 `submission.txt`
2. **容器工作空间文件** → `extract_workspace_files` 提取到 `{agent_id}/` 目录

发布者容器内的 `/evaluate_batch` 会看到：
```
/tmp/review_xxx/
  {agent_id_1}/
    submission.txt      ← 数据库内容（必定有）
    solution.md          ← 邻居写的文件（如果有）
    poem.txt             ← 邻居写的文件（如果有）
  {agent_id_2}/
    submission.txt
    ...
```

## 四、日志分析

```
[BountyHub] 提交成功: bounty_id=..., agent_id=fe7eeb23, sub_id=...
```

这说明：
1. 邻居 `fe7eeb23` 调用了 `/bounties/{id}/submit`
2. 内容存入数据库 `submissions.content`
3. **没有写入任何文件**

---

## 五、解决方案

### 方案 A：确保邻居写文件（推荐）

在 `agent_server.py` 的 `process_bounty_task` 中，处理完任务后，**必须**将结果写入工作空间文件：

```python
# 处理完任务后，写入 solution 文件
solution_file = Path(WORKSPACE_DIR) / "solution.md"
solution_file.write_text(solution_content)

# 然后提交到 BFF
async with session.post(f"{bff_url}/bounties/{bounty_id}/submit", json={
    "agent_id": CONVERSATION_ID,
    "content": solution_content,  # 同时包含文本内容
    ...
})
```

### 方案 B：改进提取逻辑

如果邻居没有写文件，可以：
1. 让邻居在提交前将工作空间文件通过 API 上传
2. 或者直接在提交内容中包含完整结果

---

## 六、当前实现的问题

即使邻居写了文件到工作空间，评审流程也可能失败，因为：

1. **文件提取时机**：邻居可能在任务完成后就清理了工作空间
2. **提取成功率**：取决于容器是否还在运行、文件是否存在

### 建议的修改

在 `process_bounty_task` 中，**同时**：
1. 将结果写入工作空间文件
2. 提交到数据库

这样评审时就有双重保障。

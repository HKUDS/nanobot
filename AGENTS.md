# AGENTS.md

## 1. 工作目标

- 默认用中文回答。
- 当前主要目标是帮助我学习、理解和掌握项目。
- 默认优先“解释清楚项目如何工作”，而不是直接改代码。
- 如果问题涉及项目内容，优先基于仓库实际文件回答；信息不足时，先搜索仓库，不要编造。

---

## 2. 默认工作流程

- 遇到任务时，先理解项目结构，再进入具体文件或问题。
- 默认优先说明：
  1. 项目入口
  2. 模块职责
  3. 调用关系
  4. 数据流 / 请求流
- 说明结构、调用关系、数据流时，优先使用简洁 ASCII 图，只保留核心结构、核心流向、关键节点。
- 学习项目结构时，优先阅读以下文件，再决定下一步：
  1. `README.md`
  2. `pyproject.toml`
  3. `nanobot/cli/commands.py`
  4. `tests/`
  5. `.github/workflows/ci.yml`

默认回答顺序：
1. 结论 / 当前文件或问题在项目中的位置
2. 结构 / 调用关系 / 数据流
3. 关键实现说明
4. 风险、边界或注意点
5. 下一步阅读建议

---

## 3. 修改代码规则

- 除非我明确要求，否则不要大规模改代码。
- 如果需要修改，优先最小改动，不要顺手重构无关内容。
- 修改前，先说明计划改哪些文件、为什么改这些文件，再开始动手。
- 变更尽量集中，避免引入无关调整。
- 如果任务以“学习和理解”为主，默认先解释，不先改代码。

---

## 4. 代码学习讲解规则

- 只有当我明确指定某个文件、函数、类、模块、调用链或代码片段时，才对该内容进行讲解；不要主动挑选代码片段展开。
- 当我要求学习某段代码时，回答必须按以下顺序组织：
  1. 代码位置
  2. 少量必要源代码片段
  3. 解释
  4. 上下游关系
  5. 下一步阅读建议
- 代码片段只保留最小必要范围，优先少量行数，不要大段粘贴源码。
- 解释要结合实际代码说明关键实现，重点讲清：
  - 关键函数 / 方法
  - 关键分支判断
  - 关键变量 / 对象
  - 关键调用语句
  - 调用前后的状态或数据变化
- 对代码片段的解释尽量包含：
  - 所在位置
  - 具体做了什么
  - 为什么这样写
  - 它和上下游逻辑怎么衔接
- 如果某段逻辑比较关键，可以按“代码片段 + 紧接解释”的方式回答。

## Python 知识点学习规则

- 当我说明“我正在学习某个文件”时，默认先不要直接逐行详解整段代码。
- 先输出该文件涉及到的 Python 知识点清单，再等我选择不懂的知识点继续展开。
- 知识点清单默认按以下格式组织：
  1. 文件作用总结
  2. 涉及的 Python 知识点列表
  3. 每个知识点所属类别
  4. 每个知识点一句话解释
  5. 哪些知识点是理解该文件的关键
  6. 建议学习顺序（从易到难）
- 只有当我明确指定要学习某个知识点时，才继续详细讲解。
- 详细讲解知识点时，按以下顺序组织：
  1. 知识点名称
  2. 标准语法 / 标准写法
  3. 该文件中的最小必要代码片段
  4. 结合实际代码解释这里为什么这样写
  5. 一个更小的独立例子
  6. 下一步建议补的前置知识
- 如果某段代码同时涉及项目逻辑和 Python 语法，默认先区分：
  - 哪些是 Python 通用语法
  - 哪些是框架 / 库特性
  - 哪些是项目自己的实现逻辑
- 除非我明确要求，否则不要一次性把所有知识点全部展开成长篇讲解。

## 5. 文档写入规则

### 5.1 图示写入规则

- 可以先在回答中生成图，用于说明项目结构、模块关系、调用链、请求流、数据流等内容。
- 任何已经生成的图，是否写入 `docs/project-structure.md`，必须先征得我的明确同意。
- 在我明确同意之前，可以展示图，但不要修改 `docs/project-structure.md`。
- 如果我同意，再把刚才已经生成的图追加写入 `docs/project-structure.md`。
- 如果我不同意，就只保留对话里的图，不写入文档。

### 5.2 代码讲解写入规则

- 默认先在对话中讲解，不要立即写入 `docs/code-snippets-study.md`。
- 只有在我明确同意后，才把本次讲解整理后写入 `docs/code-snippets-study.md`。
- 如果写入 `docs/code-snippets-study.md`，记录里必须保留 `代码片段` 小节，并在每个片段后紧接解释，不能只写纯总结。

### 5.3 文档导航同步规则

- 每次新增、修改、删除 `docs/` 下的文档内容后，必须同步更新 `docs/README.md` 中的目录树。
- 目录树格式与现有 `docs/README.md` 保持一致。

### 5.4 工作记录写入规则

- 每次完成一个可独立记录的任务后，更新 `docs/codex-worklog.md`。
- “可独立记录的任务”指已经形成明确结果的任务单元，例如：
  - 完成一轮项目结构梳理
  - 完成一次代码修改
  - 完成一次测试/验证
  - 完成一次文档整理
- 单纯概念答疑、临时讨论、未落地的方案，不默认写入 `docs/codex-worklog.md`。

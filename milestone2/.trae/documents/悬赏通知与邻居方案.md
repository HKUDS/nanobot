# 悬赏任务系统技术方案

## 一、当前问题分析

### 1.1 悬赏任务通知问题
- 问题：发布悬赏后，邻居节点不能及时收到通知
- 原因：轮询间隔 5 秒可能仍然太长，或通知创建后没有正确推送

### 1.2 完成后通知重复刷新问题
- 问题：悬赏关闭（completed）后，前端通知铃铛持续闪烁/刷新
- 原因：每次 `fetchNotifications` 检测到 `status === 'completed'` 就弹出 `ElMessage.success`，但没有标记为已提醒

### 1.3 邻居发现逻辑
- 当前：只通知 `node_relations` 表中有边连接的节点
- 问题：可能存在有边但没收到通知的情况

---

## 二、解决方案设计

### 2.1 悬赏通知机制优化

#### 问题根因
- `notify_neighbors` 创建通知后，容器轮询间隔 5 秒
- 前端也是 5 秒轮询
- 即使容器收到了通知，处理完成后 BFF 更新状态，但前端不知道要停止轮询

#### 优化方案
1. **发布悬赏时立即触发邻居容器检查**（而非等待轮询）
   - 在 BFF 中创建通知后，向所有邻居容器发送一个"强制检查"信号
   - 或让邻居容器实现 WebSocket 推送（但复杂度过高）

2. **简化方案：缩短轮询 + 完成后标记**
   - 保持 5 秒轮询（可接受延迟）
   - 前端对已完成的悬赏不再弹出重复提醒

### 2.2 通知刷新控制

#### 前端优化
```javascript
// 记录已弹窗提醒过的通知
const notifiedCompletions = new Set()

async function fetchNotifications() {
  const newNotifications = res.data.notifications || []

  for (const n of newNotifications) {
    if (n.status === 'completed') {
      // 只有首次发现完成状态时才弹窗
      if (!notifiedCompletions.has(n.id)) {
        notifiedCompletions.add(n.id)
        ElMessage.success(`任务已完成！悬赏 ID: ${n.bounty_id.substring(0, 8)}...`)
      }
    }
  }
  // ...
}
```

### 2.3 邻居通知完整性保证

#### 当前流程
1. 发布者调用 `create_bounty` 创建悬赏
2. `create_bounty` 末尾调用 `notify_neighbors(issuer_id, bounty_id)`
3. `notify_neighbors` 查询 `node_relations` 获取邻居
4. 为每个邻居创建 `notifications` 表记录

#### 确保通知到达的关键点
1. **邻居关系正确建立**：fork 时父子关系必须正确写入 `node_relations`
2. **查询邻居不出错**：`get_neighbors` 必须返回所有有效邻居
3. **通知创建不遗漏**：循环内每个邻居都必须 INSERT 成功

#### 调试日志
在 `notify_neighbors` 中添加完整日志：
```
[BountyHub] notify_neighbors: issuer=xxx, bounty=xxx, 邻居数=3
[BountyHub] 创建通知: notification=xxx, node=邻居1
[BountyHub] 创建通知: notification=xxx, node=邻居2
...
```

## 三、通知流程详细设计

### 3.1 完整时序图

```
[发布者A] --create_bounty--> [BFF]
                            |
                            v
                      [创建 bounty 记录]
                            |
                            v
                      [notify_neighbors]
                            |
                            v
                      [查询 node_relations]
                            |
                            v
                      [为每个邻居创建 notification]
                            |
              +-------------+-------------+
              |             |             |
              v             v             v
         [邻居B通知]   [邻居C通知]   [邻居D通知]
         (pending)     (pending)     (pending)
              |             |             |
              v             v             v
        [容器B轮询]   [容器C轮询]   [容器D轮询]
        /notifications/B  /notifications/C  /notifications/D
              |             |             |
              v             v             v
        [看到pending]  [看到pending]  [看到pending]
              |             |             |
              v             v             v
        [处理任务]    [处理任务]    [处理任务]
              |             |             |
              v             v             v
        [submit]       [submit]       [submit]
              |             |             |
              v             v             |
        [更新notification=completed]      |
                            |             |
                            v             v
                      [发布者A关闭悬赏]
                            |
                            v
                      [发布者容器评分]
                            |
                            v
                      [更新bounty.status=completed]
                            |
                            v
                      [前端弹窗: 任务完成]
```

### 3.2 关键检查点

| 阶段 | 检查项 | 验证方式 |
|------|--------|----------|
| 创建悬赏 | 邻居数 > 0 | BFF 日志显示 `[BountyHub] notify_neighbors: ... 邻居数=X` |
| 通知创建 | 每个邻居都有 notification | 数据库 `notifications` 表记录数 = 邻居数 |
| 容器轮询 | 能获取到自己的通知 | 容器日志显示 `[Notifications]` 且有 pending 项 |
| 任务处理 | 通知状态变为 processing | 数据库 `notifications.status = 'processing'` |
| 提交完成 | submission 记录存在 | 数据库 `submissions` 表有新记录 |
| 悬赏关闭 | bounty.status = 'completed' | 数据库 `bounties.status = 'completed'` |

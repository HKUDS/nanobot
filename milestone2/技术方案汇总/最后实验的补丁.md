## 2️⃣ 自动合并任务：基于堆段计数而非 MMU 页表

**当前 `KnowledgeManagerKM` 的 `submit_page` 确实维护了内部队列 `_queue`，并会根据队列长度触发合并。** 但问题在于：**这个队列是 KM 进程内存中的临时列表，而非来自页表或堆段文件。** 这意味着它完全依赖 Agent 主动调用 `/submit-page` 来积累，一旦 KM 容器重启，队列丢失。

**CWW 的正确计数来源应是：所有 Agent 堆段文件中 `merged=False` 的记录总数。** 这样即使 KM 重启，也能从持久化状态恢复计数。

### ✅ 修正方案

在 `KnowledgeManagerKM` 中增加一个方法，通过 BFF 的 `/heap/all-unmerged` 获取当前全局未合并记录数，并以此作为触发依据。

**修改 `_delayed_consolidation` 和 `_trigger_consolidation` 逻辑：**

```python
async def _should_trigger_merge(self) -> bool:
    """检查全局未合并堆段数是否达到阈值"""
    try:
        async with aiohttp.ClientSession() as s:
            resp = await s.get(f"{BFF_BASE_URL}/heap/all-unmerged")
            if resp.status == 200:
                data = await resp.json()
                return data.get("total_count", 0) >= self.merge_threshold
    except Exception:
        pass
    return False

async def _delayed_consolidation(self):
    await asyncio.sleep(self.merge_interval)
    if await self._should_trigger_merge():
        await self._trigger_consolidation()
```

这样 KM 的自动合并就**完全基于真实的堆段状态**，不再依赖易失的内存队列。

## 3️⃣ 页表机制：当前是否有维护活跃页？

**有，但未生效。** 在 `KnowledgeManagerKM` 中：

- `allocate_page` 会将页写入 `self.page_table`，状态为 `active`。
- `mark_pages_merged` 会将状态改为 `merged`。
- `get_active_pages` 返回所有 `active` 页。

**然而**，在 A 组脚本的 `single_agent_learning` 中，Agent 写入堆段时**确实调用了 `/knowledge-manager/allocate_page`**，所以页表理论上应有数据。但您之前实验中发现 `active_pages` 始终为空，根本原因是：

- BFF 转发 `/allocate_page` 时，可能未正确传递 `agent_id` 或请求体。
- 或 KM 容器内的 `page_table` 因进程重启而清空（每次实验前清理环境会重启容器）。

**当前架构下，页表并非必须。** 因为降级路径 `/heap/all-unmerged` 已完全够用，且更可靠（直接读取文件，无状态）。因此我建议**弃用页表，完全基于堆段文件进行合并触发和去重**。这更符合 CWW 的“无中心元数据”设计哲学。

## 4️⃣ Consolidator 与 KM 的角色边界（明确划分）

| 组件 | 职责 | 对外接口 |
|------|------|----------|
| **KM Agent 容器** | - 提供任务 Prompt（`/task`）<br>- 预置 Skill 0（`/preset-skill-0`）<br>- **后台监控全局未合并堆段数，达到阈值时触发合并**<br>- 提供统计信息（`/stats`） | `/task`, `/preset-skill-0`, `/stats` |
| **Consolidator 容器** | - 执行实际合并：拉取未合并堆段 + PublicMemory → SimHash 去重 → 原子替换 → 标记堆段为已合并 | `/execute_merge`（内部调用） |
| **BFF** | - 路由转发<br>- 聚合所有 Agent 的堆段（`/heap/all-unmerged`）<br>- 转发标记请求 | `/heap/all-unmerged`, `/agents/{id}/heap/mark-all-merged` |

**关键点**：KM **不存储任何合并业务数据**，仅作为**触发器**和**任务分发者**。Consolidator 是**无状态的合并执行器**。这种分离使得系统可水平扩展。

### P0-1: Memory Consolidation TypeError 修复不完整 (严重)

**问题描述**：
`agent_server.py` L329-337 的 Monkey Patch 修复了 `context_window_tokens` 为 None 的问题，但这个修复是浅层的：

```python
# agent_server.py L332-335
async def _patched_maybe(self, *args, **kwargs):
    if hasattr(self, 'context_window_tokens') and self.context_window_tokens is None:
        self.context_window_tokens = 128000
    return await _original_maybe(self, *args, **kwargs)
```

**根本问题**：
1. Monkey Patch 只在 `initialize_agent()` 时执行一次
2. 如果 `MemoryConsolidator` 实例在 patch 之后才被创建，patch 不会生效
3. `memory.py` L312 的检查 `self.context_window_tokens <= 0` 在值为 None 时会抛出 `TypeError`

**影响**：
悬赏任务处理过程中，如果触发了 `maybe_consolidate_by_tokens`，会导致容器崩溃。

**技术方案**：
```python
# 方案A: 在 MemoryConsolidator.__init__ 中确保默认值
class MemoryConsolidator:
    def __init__(self, ...):
        # ...
        if self.context_window_tokens is None:
            self.context_window_tokens = 128000  # DeepSeek context window

# 方案B: 修改 agent_server.py，在创建 AgentLoop 后立即 patch
async def initialize_agent():
    # ... 现有代码 ...
    # 延迟修复，确保 consolidator 已创建
    asyncio.create_task(_delayed_patch_consolidator())

async def _delayed_patch_consolidator():
    await asyncio.sleep(10)  # 等待 consolidator 创建
    await fix_consolidator(agent_loop)
```

**推荐方案**: 方案A，在 nanobot 源码层修复。

### P0-2: 悬赏任务 Solution 提取逻辑 Race Condition (严重)

**问题描述**：
`agent_server.py` L163-265 的 `process_bounty_task` 函数存在竞态条件：

```python
# agent_server.py L206-242
# 获取处理前的消息数
session_key = f"container:{CONVERSATION_ID}"
conv_session = agent_loop.sessions.get_or_create(session_key)
messages_before = len(list(conv_session.messages)) if hasattr(conv_session, 'messages') else 0

response = await agent_loop._process_message(inbound_msg)  # L210

# 从 conv_session 消息历史中获取最后一条 assistant 的回复作为 solution
solution_content = ""
conv_session = agent_loop.sessions.get_or_create(session_key)  # L215 - 重新获取
if hasattr(conv_session, 'messages'):
    messages = list(conv_session.messages)
    # ...
    for msg in reversed(messages):  # L221
        # 提取逻辑...
```

**根本问题**：
1. L208 获取 `messages_before` 但从未使用
2. L210 `_process_message` 是异步的，函数返回时会话消息可能还没完全保存
3. L215 重新获取 session 时，`session.messages` 可能还没更新到最新
4. L221 反向遍历消息时，可能取到的是上一轮对话的 assistant 消息

**影响**：
提交的 solution 内容可能是旧的或不完整的，导致悬赏任务结算时无法正确评估。

**技术方案**：
```python
async def process_bounty_task(bounty_id: str, notification_id: str = None):
    # ... 前置处理 ...

    if agent_loop is None:
        solution_content = f"自动接受任务：{bounty.get('title', '未知任务')} - AgentLoop未初始化"
    else:
        # 使用同步方式等待消息处理完成
        session_key = f"container:{CONVERSATION_ID}"
        conv_session = agent_loop.sessions.get_or_create(session_key)
        messages_before = len(conv_session.messages)

        response = await agent_loop._process_message(inbound_msg)

        # 等待消息同步完成（增加重试机制）
        for _ in range(10):
            await asyncio.sleep(0.5)
            conv_session = agent_loop.sessions.get_or_create(session_key)
            if len(conv_session.messages) > messages_before:
                break

        # 使用 get_history() 而非直接访问 messages
        messages = conv_session.get_history(max_messages=100)
        assistant_roles = {'assistant', 'model', 'ai', 'bot'}

        for msg in reversed(messages):
            role = msg.get('role', '').lower()
            content = msg.get('content', '')
            if role in assistant_roles and content and content.strip():
                solution_content = content.strip()
                print(f"[Bounty] 成功提取 solution，长度: {len(solution_content)}")
                break

        # 如果还是没提取到，直接从 response 获取
        if not solution_content and response:
            solution_content = getattr(response, 'content', '') or getattr(response, 'text', '')
```

### P0-3: 会话持久化竞态 (中等)

**问题描述**：
`loop.py` L448-449 和 L510-511 的会话保存逻辑：

```python
# loop.py L448-449
self._save_turn(session, all_msgs, 1 + len(history))
self.sessions.save(session)  # 异步保存，但调用者不等完成

# loop.py L510-511
self._save_turn(session, all_msgs, 1 + len(history))
self.sessions.save(session)
self._schedule_background(self.memory_consolidator.maybe_consolidate_by_tokens(session))
```

**根本问题**：
`self.sessions.save(session)` 是同步的，会立即写入磁盘。但如果容器在保存完成前崩溃，消息会丢失。

**影响**：
悬赏任务处理完成但容器异常退出时，solution 内容可能丢失。

**技术方案**：
```python
# 方案A: 在 process_bounty_task 中显式等待保存完成
await asyncio.sleep(0.1)  # 等待事件循环完成保存
conv_session = agent_loop.sessions.get_or_create(session_key)
# 强制同步保存
agent_loop.sessions.save(conv_session)

# 方案B: 使用文件锁确保原子写入
# 修改 session/manager.py 的 save 方法
def save(self, session: Session) -> None:
    path = self._get_session_path(session.key)
    temp_path = path.with_suffix('.tmp')
    # 先写临时文件
    with open(temp_path, "w", encoding="utf-8") as f:
        # ... 写入逻辑 ...
    # 原子替换
    temp_path.replace(path)
```

### P0-4: 重复提交检查不完整 (中等)

**问题描述**：
`bounty_hub.py` L56-64 的检查只基于 `bounty_id + agent_id`：

```python
async def submit_solution(self, bounty_id: str, agent_id: str, content: str, ...):
    with get_db() as conn:
        existing = conn.execute("""
            SELECT id FROM submissions WHERE bounty_id = ? AND agent_id = ?
        """, (bounty_id, agent_id)).fetchone()
        if existing:
            print(f"[BountyHub] 该节点已提交过，跳过: ...")
            return existing["id"]
```

**根本问题**：
如果同一个 agent 的多个容器实例同时调用，或者 notification 处理超时后重试，可能绕过检查。

**影响**：
一个节点可能对同一个 bounty 提交多个 solution。

**技术方案**：
```python
async def submit_solution(self, bounty_id: str, agent_id: str, content: str, ...):
    # 添加分布式锁或使用数据库唯一约束
    lock_key = f"bounty_submit:{bounty_id}:{agent_id}"

    with get_db() as conn:
        # 使用 INSERT OR IGNORE 配合唯一约束
        try:
            conn.execute("""
                INSERT INTO submissions (id, bounty_id, agent_id, content, skill_code, cost_tokens, created_at)
                VALUES (?, ?, ?, ?, ?, ?, ?)
            """, (sub_id, bounty_id, agent_id, content, skill_code, cost_tokens, datetime.now()))
        except sqlite3.IntegrityError:
            # 唯一约束冲突，说明已提交
            existing = conn.execute("""
                SELECT id FROM submissions WHERE bounty_id = ? AND agent_id = ?
            """, (bounty_id, agent_id)).fetchone()
            return existing["id"]
```

需要确保数据库表有唯一约束：
```sql
CREATE UNIQUE INDEX idx_submissions_bounty_agent ON submissions(bounty_id, agent_id);
```

### P0-6: Token 预算计算不考虑 Tool Results (中等)

**问题描述**：
`memory.py` L280-295 的 `estimate_session_prompt_tokens` 只估计了 `history`，没有考虑 tool results 的长度：

```python
def estimate_session_prompt_tokens(self, session: Session) -> tuple[int, str]:
    history = session.get_history(max_messages=0)  # 只获取消息历史
    # ...
    probe_messages = self._build_messages(history=history, ...)
    return estimate_prompt_tokens_chain(...)
```

**根本问题**：
Tool results 可能很长（如文件读取、搜索结果），会占用大量 token，但估算时没有考虑。

**影响**：
实际 prompt 可能超过 context window，导致 LLM 调用失败。

**技术方案**：
```python
def estimate_session_prompt_tokens(self, session: Session) -> tuple[int, str]:
    history = session.get_history(max_messages=0)

    # 计算 tool results 的额外 token
    tool_result_tokens = 0
    for msg in history:
        if msg.get('role') == 'tool':
            content = msg.get('content', '')
            tool_result_tokens += estimate_message_tokens(msg)
            if len(content) > self._TOOL_RESULT_MAX_CHARS:
                tool_result_tokens += (len(content) - self._TOOL_RESULT_MAX_CHARS) // 4

    # 现有逻辑...
    probe_messages = self._build_messages(history=history, ...)
    estimated, source = estimate_prompt_tokens_chain(...)

    return estimated + tool_result_tokens, source + "+tool_results"
```
